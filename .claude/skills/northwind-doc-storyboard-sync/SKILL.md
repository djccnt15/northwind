## StoryBoard.md 구조와 갱신 패턴

`doc/StoryBoard.md`는 동일한 화면(S-XX)의 구현 상태를 3곳에서 표현한다 — **세 곳 모두 갱신해야 한다**:

1. **1절 전체 화면 목록 표**: `| S-XX | 화면명 | ... | 🔲 |` → `✅`로 변경
2. **3절 상세 스토리보드**: `### S-XX 화면명 🔲` → `### S-XX 화면명 ✅` (헤더 텍스트의 이모지만 변경, 제목/본문은 유지)
3. **6절 미구현 화면 개발 우선순위 표**: 완료된 항목을 행에서 제거하고, 남은 항목의 우선순위 번호를 1부터 재정렬

## 코드 기준 검증 절차

`01_plan.md`의 "doc/ 영향 범위"를 그대로 신뢰하지 말고, S-XX별로 다음을 직접 확인한다:

- **프론트엔드**: `frontend/src/app/router.tsx`에 해당 화면의 라우트가 등록되어 있는가 (경로, 페이지 컴포넌트, 가드)
- **백엔드**: `02_backend_contract.md`(또는 실제 컨트롤러)에 해당 화면이 사용하는 엔드포인트가 구현되어 있는가

코드로 확인되지 않으면 ✅로 표시하지 않는다. `01_plan.md`와 실제 코드가 다르면 코드를 기준으로 판단하고 차이를 `05_doc_update.md`에 명시한다.

## PRD/EDR 갱신 판단 기준

- `doc/PRD.md`: 이번 작업으로 "명시적으로 만들지 않을 것" 항목이 변경된 경우에만 갱신
- `doc/EDR.md`: ERD에 없던 테이블/관계가 새로 추가된 경우에만 갱신
- 단순 화면 구현(StoryBoard만 영향)은 PRD/EDR 갱신 대상이 아니다 — 변경이 없으면 "변경 없음"으로 `05_doc_update.md`에 기록

## QA 결과 우선 확인

`04_qa_report.md`의 최종 판정을 가장 먼저 확인한다:
- PASS / CONDITIONAL PASS → 위 갱신 진행
- FAIL → 문서 변경 없이 `05_doc_update.md`에 "QA FAIL로 인해 보류" 기록
